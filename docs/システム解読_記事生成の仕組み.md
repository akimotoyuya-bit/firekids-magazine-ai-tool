# FIRE KIDS Magazine 記事生成システム 解読書

作成日: 2026-06-04
目的: 退職した前任者（佐々木氏）が構築した「ヴィンテージ時計SEO記事の量産システム」を解読し、引き継ぎ者がWordPress連携を進められるよう整理したもの。

> この資料は「仕組みの理解」用です。移行方針は `docs/CURSOR_HANDOVER.md`、記事ルールの正本は `CLAUDE.md` を参照してください。

## 0. 一言でいうと

**「AIに自由に書かせず、社内データの裏付けがある事実だけを、決まったフォーマットで書かせて、WordPress（m.firekids.jp）へ流し込む」** ための運用フォルダです。

- 媒体: FIRE KIDS Magazine（`https://m.firekids.jp/`）
- 親サイト（EC）: `https://firekids.jp/`
- 元の運用環境: Google Drive 共有 + VSCode の Cline（AI拡張）。現在は GitHub / Cursor へ移行中。

## 1. 全体の流れ（記事1本ができるまで）

```
① 題材を決める（data/article_plan_200.md などの記事計画）
        │
② 商品データ抽出      scripts/extract_brand.py  （Excel → JSON）
        │
③ 商品画像URL取得     scripts/fetch_images.py   （firekids.jp → CDN URL）
        │
④ AIが本文を生成       ※TXTがマスター
        │  ルール: CLAUDE.md
        │  事実 : correction_log.json → caliber_db.json → xlsx → firekids.jp内
        ▼
   articles/{BRAND}/{番号}_article_{slug}.txt  ← 完成本文（正本）
        │
⑤ 人間レビュー        scripts/review_generate.py（レビュー用md生成）
        │             修正点は data/correction_log.json に蓄積
        ▼
⑥ HTML化             generate_html_batch.py（TXT → WordParess Gutenberg HTML）
        ▼
   articles/{BRAND}/{番号}_article_{slug}.html
        │
⑦ X投稿文 + 画像生成   x_posts/ + scripts/make_x_image.py
        ▼
⑧ WordPress投稿       scripts/wp_uploader_local/（REST APIで予約投稿）
```

ポイント: **TXT が「正本（マスター）」で、HTML と X投稿はそこから生成される「派生物」** という思想で統一されています。直すときは必ず TXT を直します。

## 2. フォルダ・ファイルの役割一覧

### 記事・コンテンツ資産（壊さない）
| 場所 | 中身 | 規模 |
|---|---|---|
| `articles/{BRAND}/` | 記事本体（`.txt` 正本 / `.html` WP用） | TXT 377本 / HTML 291本 |
| `x_posts/{BRAND}/` | X（旧Twitter）投稿文 `.md` と画像 `.jpg` | md 384本 |
| `articles/{BRAND}/_posted/`, `x_posts/{BRAND}/_posted/` | 投稿済みを退避するフォルダ |  |

ブランド: BREITLING / CARTIER / CITIZEN / IWC / JLC / LONGINES / OMEGA / ORIENT / OTHER / ROLEX / SEIKO / THEME / TUDOR / UNIVERSAL / VACHERON
（THEME = ブランド横断のコラム、OTHER = 上記以外のブランド）

### ルール・仕様書（記事品質の決まりごと）
| ファイル | 役割 |
|---|---|

| `README.md` | 旧運用の全体説明 |
| `templates/cline_instructions.md` | AIに渡していた**実際の指示文（プロンプト雛形）** |
| `templates/article_html_template.html` | HTML記事の雛形 |
| `templates/factcheck_template.md` | ファクトチェックのレポート雛形 |
| `templates/x_post_templates.md` | X投稿の文面パターン集 |

### データ（事実の裏付けソース。優先順位が重要）
| ファイル | 役割 | 優先度 |
|---|---|---|
| `data/correction_log.json` | 人間（商品部）が実機確認して直した修正ログ | **1番（最優先）** |
| `data/caliber_db.json` | キャリバー仕様DB（石数・振動数・年代・Ref） | 2番 |

> 重要: **AIの一般知識で書くのは禁止**。上記で裏付けが取れない情報はそのセクションごと省略するルール。
> `correction_log.json` の値は `caliber_db.json` より優先（例: SEIKO Cal.4420 は「27石・18,000振動」が正、DBの「25石・36,000振動」は誤り）。

## 3. 記事の中身フォーマット（TXTの構造）

実例（`articles/IWC/245_article_round_case.txt`）の骨格:

```
# {タイトル}｜FIRE KIDS Magazine   ← H1（ファイル先頭）
対象ブランド: IWC
カテゴリ: IWC
生成日: 2026.03.04
---
{導入文（リード）}
---
## {セクション}   ← H2
### {サブ}        ← H3
| 表 | … |        ← Markdownテーブル
- 箇条書き
**太字**
---
## よくある質問   ← FAQ（Q/A形式）
## まとめ
**{ブランド}商品一覧を見る**
https://firekids.jp/products/list?category_id=12&utm_...  ← CTAリンク
```

必須の構成要素（`CLAUDE.md`規定）:
1. 導入文 → 2. 本文（H2/H3）→ 3.「こんな方におすすめしたい」→ 4. FAQ（3〜5問）→ 5. まとめ → 6. CTA

## 4. HTML化の仕組み（generate_html_batch.py）

TXTを WordPress の **Gutenbergブロック形式** HTMLに変換します。主な処理:

- 先頭 `# ` 行からタイトルを抽出（`｜FIRE KIDS Magazine` を除去）
- Markdown を `<!-- wp:paragraph -->` `<!-- wp:heading -->` `<!-- wp:table -->` `<!-- wp:list -->` ブロックへ変換
- `**太字**` → `<strong>`
- ブランドごとに用意した CDN画像（`BRAND_IMAGES`）を本文の H2 セクション間に3枚挿入
- FAQ の Q/A を抽出して **JSON-LD（FAQPage）** を生成
- メタ情報コメント（title/description/OG/Twitter Card）＋ Article の JSON-LD を生成
- CTAボタン（`wp-block-button`）を中盤と末尾に配置

> ⚠ 注意1: このファイルの `BASE` は Mac固定パス（`/Users/sasakitasuku/...`）。Windowsでは動かないので要修正。
> ⚠ 注意2: 旧ドキュメントに出てくる `convert.py` は**現リポジトリに存在しません**。HTML生成の実体はこの `generate_html_batch.py`。
> ⚠ 注意3: コード版のメタには `canonical_url` / `og:url` が入りますが、`HANDOVER_HTML_GENERATION.md` の最新仕様では「出力しない」。仕様とコードに差分があります。

## 5. WordPress連携（ご質問の本丸）

連携の「配線」はすでに用意されています。すべて **WordPress REST API（`/wp-json/wp/v2/...`）** 経由です。

### 投稿ツール `scripts/wp_uploader_local/app.py`（Flask, ポート8000）
- HTMLファイルをアップロード → タイトル・アイキャッチ画像・ブランドを自動抽出
- タグ自動作成、カテゴリ（「時計の基礎知識」「コラム」）、投稿者（編集部に固定）
- 予約投稿（空欄なら即時公開）
- 認証は **Application Password** を `.env` に設定（`WP_USER` / `WP_APP_PASSWORD` / `WP_BASE_URL`）
- 接続確認は `/health`（`/wp-json/wp/v2/users/me` を叩く）

### 非公開化ツール `scripts/wp_unpublisher_local/app.py`（ポート8001）
- 記事URLを貼り付けて `private` / `draft` に一括変更

### Next.js版の dry-run `src/app/wordpress/page.tsx`
- 投稿前に内容を確認する画面まで実装済み。**本番投稿ボタンは未実装**。

### 連携に必要なもの（現状欠けているもの）
- `.env` 実体（認証情報）は含まれていない → WP管理画面で Application Password を新規発行して設定が必要
- 依存パッケージ: `pip install -r scripts/wp_uploader_local/requirements.txt`（Flask, requests, python-dotenv）

## 6. 品質ルール（特に重要な禁止事項）

`CLAUDE.md` と `src/lib/validation.ts`（自動検出ロジック）で機械的にもチェックされます。

| 禁止 | 内容 |
|---|---|
| FK番号 | `FK000002` 等の内部管理番号を本文に出さない |
| 価格 | 相場・販売価格を書かない（変動するため） |
| 個別商品URL | `/products/detail` 等の個別ページを貼らない（CTAはカテゴリ一覧のみ） |
| 外部ソース | Wikipedia/Ranfft等を事実として採用しない |
| 擬似エピソード | 「店頭でよく聞かれる」等のAIが想像した接客話を書かない |
| 強い断定/店員視点 | 「断言します」「時計屋として」等の語気を使わない |
| 根拠なき紐付け | 職業×特定モデルを根拠なく結びつけない |

CTAリンクには必ず UTM を付与:
- 記事: `utm_source=firekids_magazine&utm_medium=seo&utm_campaign=organic`
- X投稿: `utm_source=x&utm_medium=social&utm_campaign=magazine`

> 補足: 既存記事の一部（例: IWC 245）は古い UTM（`utm_source=magazine&utm_medium=article...`）のまま残っており、ルールと不一致。再投稿時は要確認。

## 7. AIへの指示方法（前任者の運用）

前任者は `templates/cline_instructions.md` の指示文を Cline に貼り付けて記事を作らせていました。指示文には「3段階ファクトチェック（caliber_db → xlsx → firekids.jp内）」「禁止事項チェック」「TXT+HTML出力」「メタ/JSON-LD生成」までが含まれています。

つまり **「プロンプト＋ルールブック＋データDB」で品質を担保する** 設計で、AIの創作余地を意図的に絞り込んでいるのが特徴です。

## 8. 引き継ぎ上の注意点まとめ

1. `generate_html_batch.py` の `BASE` パスがMac固定 → Windows用に要修正。
2. `convert.py` は存在しない（HTML生成は `generate_html_batch.py`）。
3. `.env`（WP認証）は無いので新規発行が必要。
4. `factcheck_logs/`, `review_docs/`, `data/snapshots/`, `data/images/` は一部現物が欠けている可能性あり。
5. HTML仕様は `HANDOVER_HTML_GENERATION.md`（最新）とコードに差分あり。実装時は仕様書を優先。
6. TXTがマスター。HTMLを直接編集しない。
7. 既存 `articles/` `x_posts/` `data/` は削除・移動しない。

## 9. 「WordPressとつなげる」次の一手（候補）

- A. 既存の `wp_uploader_local` を Windows で動かして、HTML記事を実際に投稿できる状態にする（最短）
- B. Next.js管理画面に本番投稿機能を実装し、ブラウザ上で選択→投稿まで完結させる
- C. まず接続テスト（`/health`）だけ通して、APIで繋がることを確認する

いずれも **Application Password の発行と `.env` 設定** が最初の関門です。
